## ADR-001: gunicornのworkers=1固定

**日付:** 2026-05-04  
**状態:** 採用中

### 背景

本アプリは `app.py` のモジュールレベルに以下の辞書を持つ:

```python
original_names = {}
```

この辞書は、アップロード時に元のファイル名（ユーザーが付けたExcelファイル名）を
`session_id` をキーとして格納し、ダウンロード時に参照してファイル名を復元する用途で使う。

### 問題

gunicornで `--workers 2` 以上にすると、各ワーカープロセスが独立したメモリ空間を持つため、
`original_names` 辞書がプロセス間で共有されない。

- アップロードを処理したプロセス（ワーカーA）が `original_names[session_id] = "元のファイル名.xlsx"` を書く
- ダウンロードリクエストをワーカーBが受け取ると、ワーカーBの `original_names` にはそのキーが存在しない
- 結果: ダウンロード時のファイル名がデフォルト値（`output.xlsx`）になる

### 選択肢と却下理由

| 案 | 内容 | 却下理由 |
|----|------|----------|
| **A: workers=1固定（採用）** | 現状維持。プロセスは常に1つなので辞書共有が保証される | — |
| B: Redis/Memcachedで辞書を共有 | マルチワーカー対応になる | Renderの無料プランでは外部キャッシュサービスの追加コストが発生する。単一ユーザーの社内ツールでスケールアウトの必要性がない |
| C: ファイルシステムに`.meta.json`として保存 | `{session_id}.meta.json` にファイル名を書き出す | 将来の改善案として有効だが、現状の動作に問題がないため過剰な変更になる |
| D: Flaskセッションに格納 | セッションクッキーにファイル名を持たせる | セッションクッキーはブラウザ側に保存されるためサイズ制限があり、複数ファイル管理が複雑になる |

### 決定

**案A（workers=1固定）を採用する。**

`render.yaml` の起動コマンドを `--workers 1` で固定し、変更を禁止する。

```yaml
startCommand: gunicorn app:app --worker-class gevent --workers 1 --timeout 300 --bind 0.0.0.0:$PORT
```

### トレードオフ

| 項目 | 影響 |
|------|------|
| 並行処理能力 | geventによるI/O多重化で補完しているため、単一ユーザーの社内ツール用途では十分 |
| スケールアウト | workers>1にした瞬間にファイル名バグが再発する（この制約を将来の開発者が知らない場合） |
| 変更コスト | 案Cへの移行は比較的低コストだが、現状の必要性がない |

### 影響範囲

- `render.yaml` の `startCommand`
- ローカル開発時はFlask開発サーバーを使うため影響なし
- `--workers` を変更する場合は必ず `original_names` の代替手段を実装してから行うこと

## ADR-002: Gemini APIをHTTP直接呼び出し（公式SDKを使わない）

**日付:** 2026-05-04  
**状態:** 採用中

### 背景

Gemini APIへのリクエストは `google-generativeai` 公式SDKではなく、`requests` ライブラリによるHTTP直接呼び出しで実装している。

### 決定理由

- geventのモンキーパッチ適用範囲を制御しやすい
- SDKのバージョン変更に依存しない
- APIキーをHTTPヘッダー（`x-goog-api-key`）経由で送信することで、URLパラメータへの漏洩を防いでいる

### 却下された案

| 案 | 却下理由 |
|----|----------|
| `google-generativeai` SDK | SDKの内部HTTP実装がgeventと競合するリスク。モンキーパッチ後のソケット動作が不透明 |

## ADR-003: 二重API呼び出し防止のための processing_sessions フラグ

**日付:** 2026-05-04  
**状態:** 採用中

### 背景

`/process/<session_id>` は SSE ストリームエンドポイントで、処理中に Gemini API を呼び出す。  
ブラウザのリロード・複数タブ開き等で同一 `session_id` に対して並行リクエストが来た場合、  
gevent がコルーチンを両方実行し、Gemini API が2回呼ばれてしまう（コスト2倍・30枚×2回の画像送信）。

### 選択肢と却下理由

| 案 | 内容 | 却下理由 |
|----|------|----------|
| **A: モジュールレベル `set()` で管理（採用）** | `processing_sessions = set()` でフラグ管理。2回目は即 409 を返す | — |
| B: DBやファイルにフラグを書く | SQLite等のファイルベースのロック | workers=1固定（ADR-001）の環境ではオーバーエンジニアリング |
| C: Redis/Memcachedで分散ロック | マルチプロセス対応のロック | ADR-001と同様、Renderの無料プランで外部サービスの追加コストが発生。単一ユーザー用途で不要 |
| D: クライアント側でボタン無効化のみ | 2回目のクリックを防ぐ | HTTP直接アクセスや複数タブは防げない。サーバーサイドのガードが必要 |
| E: `threading.Lock()` | ロックオブジェクトで排他制御 | gevent + workers=1 の協調スケジューリング環境では `set` の確認・追加が yield を挟まないため、ロックなしで十分 |

### 決定

**案A（モジュールレベル `set()` によるフラグ管理）を採用する。**

`processing_sessions.add()` は Flask ルートハンドラ内（ジェネレータ実行前）で同期的に呼ばれるため、  
gevent の協調スケジューリング下でも2番目のリクエストが確実に 409 で弾かれる。  
`discard()` は要素が存在しなくても例外を出さないため、二重削除に耐性がある。

### フラグ解放のパターン（実装上の注意）

`process()` 内には `generate()` を通らない早期 `return` パスが存在するため、  
フラグ解放は2通りの方法を使い分ける:

| パス | フラグ解放方法 |
|------|---------------|
| `generate()` に到達するパス（正常・例外問わず） | `generate()` 内の `finally: processing_sessions.discard(session_id)` |
| `add()` 後の早期 `return`（テンプレート未検出等） | `return` の直前に `processing_sessions.discard(session_id)` を明示的に呼ぶ |

`add()` より前の早期 `return`（セッション未存在の 404、重複検出の 409）には `discard()` は不要。  
今後 `add()` より後に早期 `return` を追加する場合は、必ず `discard()` を同時に追加すること。

### トレードオフ

| 項目 | 影響 |
|------|------|
| workers=1 への依存 | `processing_sessions` はプロセスメモリ上のため、workers>1 にすると各プロセスが独立したセットを持ち、フラグが機能しなくなる。ADR-001（workers=1固定）と連動した制約 |
| workers>1 への移行時 | `processing_sessions`（本ADR）と `original_names`（ADR-001）の両方を Redis 等の外部ストレージに同時に移行する必要がある |
| ジェネレータ未消費時 | SSE 接続が確立前にクライアントが切断した場合、`finally` が実行されずフラグが残るリスクがある。ただし SSE エンドポイントは接続確立後すぐにジェネレータが消費されるため実用上の問題はない。Render のコンテナ再起動でリセットされる |

### 影響範囲

- `app.py` のモジュールレベル変数とロジックのみ
- ADR-001（workers=1固定）と連動。`--workers` を変更する場合は `original_names`（ADR-001）と `processing_sessions`（本ADR）の両方の代替手段を同時に実装すること
